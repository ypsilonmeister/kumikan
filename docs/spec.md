## 1. アプリケーション概要

本アプリケーションは、同一のローカルネットワーク（LAN）内において、家族で机を囲みながら各々の端末（スマートフォン、タブレット、PC）を持ち寄って遊べる、ブラウザ完結型のリアルタイム漢字パズルゲームです。WebRTC技術を活用し、外部サーバーへの依存を最小限に抑えた完全P2P通信モデルを採用します。

---

## 2. ゲームルール定義

* **初期準備:** 各プレイヤーにランダムな漢字パーツ（へん、つくり、かまえ等）が一定数（$N$枚）配られます。残りのパーツは共通の「山札」として伏せられます。
* **ターン進行:**
* 手番のプレイヤーの開始時に、山札からパーツが1枚自動的にめくられ、全員の画面の「場」に共有表示されます。
* 手番プレイヤーは、その場札に対して合体可能なパーツを自分の手札から選択して提示します。


* **判定と分岐:**
* **正解（漢字成立）:** 組み合わせが正しい場合、その漢字が完成。完成した漢字はプレイヤーの得点（獲得山）となり、プレイヤーはルールに従って山札からカードを補給、または続けて行動します。
* **不正解・パス:** 合わせられるパーツがない、または間違えた場合は何も獲得できず、次のプレイヤーへ手番が移ります。


* **終了条件:** いずれかのプレイヤーの手札が0枚になった時点でゲーム終了。最も多くの漢字を完成させた（得点の高い）プレイヤーの勝利となります。

---

## 3. システムアーキテクチャ & P2P接続仕様

外部サーバーの仲介や生存リスクに依存しない「完全サーバーレスP2P」を実現するため、WebRTC DataChannelのシグナリングプロセスをQRコード経由で手動最適化する方式を採用します。

### 3.1. シグナリングデータの軽量化（SDP Munging）

通常のWebRTCで生成されるSession Description Protocol（SDP）は3KB〜6KBと肥大化しやすく、QRコードでの読み取りが困難になります。本システムでは音声・映像メディアセクションを最初から排除（`createDataChannel`のみ）し、さらに以下の文字数削減ロジック（SDP Munging / Minimizing）を適用します。

* **不要行の削除:** `a=rtpmap`、`a=fmtp` などのメディア用コーデック関連行、予備の暗号化プロファイル、不要なネットワーク経路（Candidate）を手作業または正規表現で間引きます。
* **必須パラメータのJSON抽出:** 接続確立に絶対必要な以下の4要素のみをSDPから抽出し、オブジェクト化します。
* `ice-ufrag` (ユーザー名フラグメント)
* `ice-pwd` (パスワード)
* `fingerprint` (証明書ハッシュ)
* `candidate` (最優先のローカルIPv4ホスト経路のみ)



### 3.2. QRコードペイロード最適化フロー

抽出したオブジェクトのキー名を1文字に圧縮し、シリアライズした文字列をBase64または軽量な圧縮アルゴリズム（LZMA等）にかけることで、ペイロードサイズを数百バイト（300〜500文字）に抑えます。これにより、QRコードのセル密度をバージョン5〜7（37×37〜45×45ドット）に維持し、スマホカメラでの一瞬でのスキャンを可能にします。

> **【重要】LAN内接続におけるICE Candidateの選定**
> STUN/TURNサーバーを使用しないローカル環境のため、ブラウザが列挙するネットワーク経路から「192.168.X.X」などのプライベートIPv4アドレス（ホスト候補）を確実に最優先として残すロジックを実装します。

## 4. データ構造と判定ロジック設計

### 4.1. 組み合わせ辞書データ（KANJI_RECIPES）

順序に依存しない判定を行うため、パーツの組み合わせキーはアルファベットまたはUnicode順でソートして保持します。

```javascript
// 判定用データ構造の例
const KANJI_RECIPES = {
  "木,目": "相",
  "可,さんずい": "河",
  "言,吾": "語",
  "月,木": "棚"
};

// 判定関数
function checkCombination(partA, partB) {
  const key = [partA, partB].sort().join(',');
  return KANJI_RECIPES[key] || null; // 存在すれば完成漢字、なければnull
}

```

### 4.2. P2P通信パケット構造（データチャンネル）

ゲーム中の状態同期は、JSON形式のシリアライズメッセージをDataChannel経由でブロードキャストします。

| イベントタイプ (`type`) | 送信データ例 (`payload`) | 説明 |
| --- | --- | --- |
| `GAME_START` | `{ turnOrder: [0, 1, 2], handSize: 5 }` | 親機から全子機へ、ゲーム開始と順番を通知 |
| `DRAW_FIELD` | `{ part: "木" }` | 山札からめくられた場札のパーツ情報を同期 |
| `SUBMIT_ANSWER` | `{ playerId: 1, part: "目", result: "相" }` | 手番プレイヤーが正解したことを全体に通知 |
| `NEXT_TURN` | `{ nextPlayerId: 2 }` | 手番の移行を通知 |

## 5. UI/UX ＆ グラフィック方針

* **直感的な操作性:** スマートフォンやタブレットでのプレイを大前提とし、場のカードに対して自分の手札を重ねるようなタップ＆ドラッグ操作を主軸とします。
* **視認性と「余白」の重視:** 複雑になりがちな漢字パーツをすっきりと見せるため、文字フォントは大きく太く設定。カードの境界線や背景には過度な装飾を避け、十分なマージン（空気感）を持たせたモダンでクリーンな画面設計を徹底します。
* **合体演出:** パーツ同士が結合して1つの漢字に変化した瞬間には、パーツが中央に吸い寄せられてフラッシュするような、子供が達成感を得られるアニメーション演出をCSS Transition/Animation等で実装します。
